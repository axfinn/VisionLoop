# DEFINE - 问题定义

## WHY - 为什么要做？

### 痛点分析
1. **现有方案缺失**：家庭/小型商业场景缺乏低成本、易部署的本地化智能监控录像系统。Frigate等开源方案依赖复杂配置和Linux环境，不适合普通用户。
2. **云端方案弊端**：云存储有隐私泄露风险、持续订阅费用高、网络依赖强，断网即失效。
3. **事件检测孤岛**：传统NVR的移动侦测误报率高，无法识别摔倒、哭声等高价值事件，导致用户需人工盯屏或事后回查，效率极低。
4. **回放体验落后**：多数方案不支持HTTP Range拖拽，回放加载慢，体验差。

### 如果不做会怎样？
- 用户被迫选择高成本云方案或低效人工盯控
- 摔倒等紧急事件无法及时发现，尤其影响老人/婴幼儿监护场景
- 历史录像回放仍需完整下载，无法快速定位

## WHO - 谁受益？

### 主要用户
| 用户 | 场景 | 最关心 |
|------|------|--------|
| 家庭用户（婴幼儿/老人监护） | 宝宝监控、老人防摔 | 事件及时推送（哭声、摔倒）、隐私本地存储 |
| 小型商业（店铺/办公室） | 安防监控 | 陌生人闯入检测、录像长期存储、回放便捷 |
| 技术能力有限的用户 | 开箱即用 | 双击运行、浏览器访问、无需配置 |

### 用户核心诉求优先级
1. **可靠性** - 录像不丢失、分段完整
2. **低延迟** - 监看延迟<1秒，事件实时推送
3. **低资源** - CPU占用低（<5%）、静默运行
4. **易用性** - 无需配置、直接使用
5. **可回溯** - 事件可查、录像可拖拽

## WHAT - 要做什么？

### 交付物清单

1. **Gin HTTP服务器**（:8080）
   - `GET /api/clips` - MP4文件列表（JSON）
   - `GET /api/clips/:name` - MP4文件Range拖拽服务
   - `GET /api/events` - 事件列表（JSON，含时间戳/截图路径/类型）
   - `GET /api/ws/signal` - WebRTC信令WebSocket
   - `GET /` - Vue3 SPA静态文件服务

2. **双路编码录制系统**
   - 录制路：4Mbps H.264 → MP4分段（5分钟/文件）
   - 监看路：500kbps H.264 → Pion WebRTC zerolatency推送
   - 硬件编码优先级：h264_qsv → h264_nvenc → libx264

3. **事件检测引擎**
   - 异步抽帧（500ms间隔）
   - YOLOv8姿态检测 → 摔倒
   - 音频频谱分析 → 哭声/异响
   - YOLOv8人形+人脸 → 陌生人闯入
   - 检测结果：时间戳 + 截图 → 事件库

4. **存储GC系统**
   - 文件命名：`2026-03-25_14-30-00.mp4`
   - 磁盘容量检测 + 自动删除最旧文件

5. **Vue3前端**
   - 实时监看页面（WebRTC + 事件气泡标注）
   - 历史回放页面（video.js + Range拖拽 + 事件时间轴标注）
   - 事件中心（列表/截图/快速跳转回放）
   - 设置页面（存储阈值/检测开关/灵敏度）

6. **Windows分发包**
   - `VisionLoop.exe`（嵌入前端静态资源 + 模型文件）
   - `ffmpeg.dll`（含QSV/NVENC支持）
   - 双击即用，访问 `http://localhost:8080`

### 成功标准（可量化）

| 指标 | 目标值 | 验证方法 |
|------|--------|----------|
| 监看延迟 | < 1秒（端到端） | 浏览器实测 |
| 编码CPU占用 | < 5%（硬件编码） | 任务管理器 |
| 录制完整性 | 0丢帧（无缓冲channel策略） | 日志验证 |
| 事件检测召回率 | 摔倒>80%、哭声>85%（模型评估集） | 测试集验证 |
| Range拖拽响应 | < 500ms | HTTP请求计时 |
| 打包体积 | < 200MB（含FFmpeg+模型） | 文件大小 |
| 启动时间 | < 5秒（双击到监看画面） | 人工计时 |

## CONSTRAINTS - 约束条件

### 技术限制
1. **gocv + FFmpeg CGO符号冲突风险** - 需验证兼容性，可能需分进程或静态编译
2. **Windows硬件编码依赖** - QSV仅Intel集显有效，NVENC需NVIDIA独显，libx264为兜底
3. **YOLOv8 ONNX算子覆盖** - onnxruntime-go需验证支持全部所需算子
4. **WebRTC P2P局限** - 目前设计为WebRTC直连，无TURN中继，网络穿透可能失败
5. **音频检测库未定** - 频谱分析具体实现方案未确定

### 资源限制
1. **开发机无GPU** - 硬编码和模型推理测试受限
2. **无目标摄像头** - 采集测试依赖虚拟摄像头或本地视频文件
3. **工期约束** - 单人开发，需优先交付核心链路

### 奥卡姆剃刀 - 最简方案
**最小可行产品（MVP）路径**：
1. 先实现软编码（libx264）+ 录像录制 + Gin文件服务 → 验证核心链路
2. 再叠加WebRTC监看
3. 最后叠加事件检测（YOLOv8可先用公开模型）
4. 硬件编码和音频检测作为后续迭代

## CORE PROBLEM - 一句话核心问题

**为家庭用户和小商户解决本地智能监控需求，使其能在低硬件占用、低延迟的前提下，实现摔倒/哭声/陌生人闯入等高价值事件的实时检测与录像回溯，同时保证隐私（数据不出设备）和开箱即用的易用性。**
